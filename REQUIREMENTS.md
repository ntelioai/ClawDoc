# mdown — Markdown & HTML Document Browser

## Purpose

A local document browser for the `Business/` workspace that makes the hundreds of `.md` and `.html` files generated by AI assistance navigable, searchable, and cross-linkable. It replaces ad-hoc file-tree exploration with a Google Drive-like UI tailored to a Markdown-heavy knowledge base.

## Problem

- AI assistance produces a steady stream of `.md` documents (proposals, plans, research, feasibility studies) and occasional polished `.html` decks/exports.
- The files live across many folders (`Business Development/`, `Commercial/`, `Products/`, `Research/`, `ProBono/`, etc.) following the `YYYY-MM-DD_project-name_document-type.md` convention.
- Finding a specific document, recalling what exists for a given prospect/product, and following references between documents is increasingly tedious.
- The Finder / IDE file tree is not designed for content-aware browsing (no titles, no previews, no search across body text, no link graph).

## Goals

1. **Discoverability** — see at a glance what documents exist, where, and what they are about.
2. **Navigation** — browse folder-by-folder like Google Drive, with the folder tree always visible.
3. **Search** — fast keyword search by filename, folder, and document body/title.
4. **Reading** — render Markdown and standalone HTML in-app with good typography.
5. **Cross-linking** — follow links between documents (relative paths in Markdown and HTML) without leaving the browser.
6. **Zero-friction** — single-command launch, works offline against the local filesystem, no external service.

## Non-Goals

- Editing documents (read-only browser; editing stays in the IDE).
- Cloud sync, multi-user collaboration, or any server-side component beyond a local dev server.
- Replacing the existing `Utils/Presentation` deck framework — `mdown` only displays/links to its outputs.
- Full-text indexing of binary files (PDFs, images) in v1.

## Users

Single user: the workspace owner, running locally on macOS.

## Scope

### Document types

- **In-scope**: `.md`, `.markdown`, `.html`, `.htm` anywhere under `Business/` (excluding `archive/`, `node_modules/`, `.git/`, and any user-configured ignore list).
- **Surfaced but not rendered**: `.pdf`, images — shown in folder listings with an icon, opened in the OS default app on click.
- **Ignored**: everything else.

### Index

- A separate **index generation step** (CLI script) walks the filesystem and produces a JSON index consumed by the UI.
- Index contains, per document:
  - Absolute and workspace-relative path
  - Folder path (for the tree)
  - Filename
  - Title (first `# heading` for Markdown; `<title>` for HTML; filename fallback)
  - Date (parsed from `YYYY-MM-DD_` filename prefix when present; otherwise file mtime)
  - Project/customer slug (parsed from filename convention when present)
  - Document type slug (parsed from filename convention when present)
  - Size, mtime
  - Body text (plain-text extract for search; capped per-file to keep index small)
  - Outbound links (relative paths referenced from the document)
- The index is regenerated on demand (manual run) and on a watch mode (optional, v2).
- Index lives at `Utils/mdown/index.json` and is git-ignored.

### UI layout (Google Drive-like)

- **Left pane — Folder tree**
  - Hierarchical tree rooted at `Business/`.
  - Expand/collapse folders; remember expansion state across reloads.
  - Folder name shows document count badge.
  - Keyword filter box at the top filters the tree to folders/documents matching the query (name + title + body, with match highlighting).
- **Right pane — Folder/document view**
  - When a folder is selected: a list/grid of documents and subfolders inside it. Each row shows title, filename, date, type, size. Sortable by name/date/type.
  - When a document is selected: rendered content.
    - Markdown rendered with GitHub-flavored syntax (tables, task lists, fenced code, footnotes).
    - HTML rendered as-is in a sandboxed iframe.
    - Relative links to other `.md`/`.html` files are intercepted and open inside the browser (updating the URL/selection); external links open in the system browser.
    - Image references resolved relative to the document's folder.
  - Breadcrumb at the top of the right pane reflecting the current path; clicking a segment navigates to that folder.
- **Top bar**
  - Global search input (searches across the whole index by title + body + filename + folder).
  - "Reindex" button to re-run index generation without leaving the UI.
  - Current document path (copyable).

### Search

- **Tree filter** (left pane): narrows the visible folder tree as you type.
- **Global search** (top bar): returns a ranked result list across the workspace with snippet previews; clicking a result opens the document with the match scrolled into view.
- Search is case-insensitive, supports multi-word AND matching, and ranks title/filename matches above body matches.
- Latency target: <100ms for typical queries against an index of a few thousand documents.

### Cross-document linking

- Markdown `[text](relative/path.md)` and `<a href="relative/path.html">` resolve against the document's folder and open in-app.
- Anchor links (`#section`) scroll within the rendered document.
- Broken links are visibly marked (e.g., red underline + tooltip showing the missing path).
- A document's "backlinks" (other documents that link to it) are listed in a side panel or footer for the current document.

### Keyboard

- `/` focus global search
- `Cmd/Ctrl+P` quick-open by filename (fuzzy)
- `Cmd/Ctrl+K` focus tree filter
- `Esc` clear current search/filter
- Arrow keys to navigate result lists and folder tree

## Architecture

- **Static-ish local web app**: Node.js script generates the index; a lightweight local server (or a single `index.html` consuming `index.json` directly via `file://` where feasible) serves the UI.
- **Stack** (proposed, open to revision):
  - Index generator: Node.js script using `gray-matter`, `marked`/`remark`, `cheerio` for link/title extraction.
  - UI: plain HTML/CSS/JS with a small framework or no framework, mirroring the lightweight approach of `Utils/Presentation`. Rendering via `marked` (Markdown) and DOMPurify (HTML sanitization where appropriate).
  - Search: in-memory index built from `index.json` on load (e.g., MiniSearch or a hand-rolled inverted index — must stay client-side and dependency-light).
- **No build step** preferred; if one is needed, keep it to a single `npm run build`.

## Commands / Workflow

- `node Utils/mdown/index.js` — regenerate `index.json` by walking the workspace.
- `node Utils/mdown/serve.js` — start the local server (or `open Utils/mdown/launcher.html` if the app can run without a server).
- The browser UI itself exposes a "Reindex" button that runs the index step (via a tiny local endpoint when the server is running).

## Configuration

- `.mdownignore` at `Utils/mdown/.mdownignore` (gitignore-style patterns) for folders/files to exclude from the index. Defaults exclude `archive/`, `node_modules/`, `.git/`, `Downloads/` (configurable).
- A small `config.json` (or top of `index.js`) for: workspace root, included extensions, max body-snippet size per file, index output path.

## Performance

- Workspace currently has hundreds of documents; design should comfortably handle ~5,000 files.
- Index generation should complete in under ~5s on a warm filesystem.
- Initial UI load (index parse + render) under ~1s for the expected scale.

## Open Questions

1. Server vs serverless: is a tiny local server (for reindex-on-click and clean URLs) acceptable, or must this run purely from `file://`?
2. Should the index include front-matter (YAML) when present, and surface fields like `tags`, `status`, `customer`?
3. Backlinks panel in v1, or defer to v2?
4. Should `.pdf` content be extracted for search (would require a PDF text-extraction step), or remain icon-only?
5. Theme: light only for v1, or include a dark theme to match the IDE?

## Success Criteria

- Opening `mdown` and finding any known document by title or customer name takes under 5 seconds.
- Cross-document links in proposals/research render and resolve without breakage.
- The user reaches for `mdown` instead of Finder/IDE tree when looking for an existing document.
